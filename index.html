Background on JAMstack

To understand JAMstack's role today, we need to:

  • Take a look at history of web pages
  • How we got here

The Early 1990s

Users demand was simple

  • Web pages initially created as document management system.
  • Web pages were hand coded in HTML containing links to other pages.
  • Initially, browsers did not have multimedia support.

Mid 1990s - Early 200s

Changing users demand

  • Web pages are not hand-coded but are generated dynamically by programming languages.
  • Web pages require application server/language runtime.
  • With dynamic sites, server would get content from databases and render web pages in real-time for each user, instead of building html pages ahead of time.
  • May Require Databases

Early Examples

  • Amazon in 1994
  • Yahoo in 1994

Content Management Systems Popularity and Blogging Software Proliferation

  • LiveJournal: 1999
  • Blogger: 1999
  • MoveableType: 2001
  • Drupal: 2001
  • MediaWiki: 2001
  • WordPress: 2003
  • Joomia: 2005
  • Magento: 2009

Dynamic Web Page - Client Side

  • JavaScript in 1995
  • DHTML in 1997: Use of JS to change the look of the page - showing/hiding elements, animations.
  • AJAX in 2006: Use of JS to retrieve and send data in the background without reloading the page.

In Summary, Dynamic Sites:

Pros Cons
Content is updated in real time. More Software, more complexity, more maintenance
Content can be customized for each user

Mid 200s to Present

Developers Demand Easier Process

Static Site Generators, SGG

  • A program/script that would take content stored in a format like XML and apply a template to that content to build static html files.
  • Has existed since the 1990s: MovableType had an option where you can build a static type.
  • Became more popular in the 2000s due to the invention of lightweight markup languages.
    • reStructuredText in 2002
    • Textile in 2002
    • markDown in 2002

Examples of SSGs

  • Jenkyll in 2008
  • Pelican in 2010
  • Hugo in 2013

Static Sites Generators:

Pros Cons
Simple; less software, less complexity Limitation on the number of authors
Limitation on the number of content
